# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## State of the repo

This repo is currently **empty of application code** — it contains only a `docs/` folder with three handoff bundles. There is no `package.json`, no `src/`, no build/lint/test tooling yet. The first real task here is almost always to **scaffold the app** per the specs below; there are no existing commands to run.

Once scaffolded (per `ARCHITECTURE.md`'s suggested stack — Vite + React 18 + TS strict + Vitest), add the standard `npm run dev` / `build` / `lint` / `test` scripts and document the actual commands here so future sessions don't have to rediscover them.

## What this project is

**RestoOS** — a multi-tenant restaurant management system frontend (React + TypeScript), covering: Menu, Recipes, Inventory, Guests (loyalty CRM), Billing (POS/Factor), PublicMenu (live price board / guest QR menu), and Analytics. Desktop back-office + tablet/phone for counter & guest-facing screens.

## The three handoff bundles — read in this order

1. **`docs/design-to-backend-handoff/`** — original lo-fi wireframes (`Wireframes.html` + `js/screen-*.jsx`) showing screen inventory and A/B/C "approach" alternatives. Historical context only — a chosen direction was picked and refined into bundle 2. Don't build from this; it predates both the hi-fi design and the real backend contract.

2. **`docs/design-to-frontend-handoff/handoff_react_ts/`** — **the primary build spec**. Contains:
   - `CLAUDE.md` — product/domain overview, locale & money decisions, non-negotiables (TS strict, modular DDD, repository pattern, pure domain math, a11y).
   - `ARCHITECTURE.md` — the suggested stack, the **module layering convention** (`domain/ → application/ → infra/ → ui/` with barrel-only cross-module imports), full folder structure, design tokens, and the **build order** (scaffold → menu → inventory → recipes → billing → guests → publicMenu → analytics → swap mock infra for HTTP).
   - `SCREENS.md` — screen-by-screen spec: domain, reads/writes, components to build, states.
   - `design/Restaurant System — Hi-Fi.html` — the chosen visual reference (open in a browser; 12 named frames, one approach per screen). Tokens here are the source of truth for styling.

3. **`docs/backend-to-frontend-handoff/`** — **the real backend API contract** (the backend has since been built in Go). Contains `README.md` (conventions: tenancy header, money, pagination envelope, error shape, dates), `api-reference.md` (every endpoint + JSON shapes + business rules), and `types.ts` (DTOs ready to copy into `api/types.ts`, generated by hand from `internal/domain/*/{request,response}.go`).

## ⚠️ Where the bundles disagree — bundle 3 (the live backend) wins

The design-spec bundle (#2) was written *before* the backend existed and contains assumptions that the actual backend (#3) overrides. When wiring `infra/*.http.ts` repositories and domain types, **trust `api-reference.md` / `types.ts`**, not the design spec's `CLAUDE.md`/`ARCHITECTURE.md` domain model. Known conflicts:

- **Money**: design spec says "Currency: Toman, integers, no decimals". The real API returns **integers in minor currency units** (e.g. cents — `1250` = `12.50`), and currency/tax-rate rules are explicitly called out as still-open/provisional. Build `shared/money` against the real API shape, not the Toman assumption.
- **Pagination**: design spec's domain model shows plain arrays everywhere. The real API paginates `GET /admin/restaurants`, `/stock-ins`, `/guests`, `/factors` with a `PaginatedResult<T>` envelope (`items/page/pageSize/totalItems/totalPages`); categories/recipe/low-stock stay plain arrays.
- **Factor status**: design spec uses `'open' | 'held' | 'paid'`; the real API uses `"unpaid" | "paid"` with no "held" state, and edits/deletes are blocked (`409`) once paid.
- **Guest tiers**: design spec uses `'New' | 'Silver' | 'Gold'`; the real API uses `"member" | "silver" | "gold" | "platinum"` with documented auto-promotion thresholds (5/15/30 visits).
- **Tenancy**: not mentioned in the design spec at all — every tenant-scoped request needs `X-Restaurant-ID: <uuid>` (there's also a non-tenant-scoped platform-admin surface at `/admin/restaurants` for onboarding/suspending restaurants, which has no corresponding screen in the design spec).
- **Server-resolved values**: factor lines are sent as `{ foodId, qty }` only — the server resolves `unitPrice` and computes `subtotal`/`total`; don't replicate that math client-side for anything but live preview.

## Architectural non-negotiables (from the design spec — still apply)

- **TypeScript strict, no `any`.** Domain types live in each module's `domain/`, never in components.
- **Modular DDD**: each feature module (`menu`, `recipes`, `inventory`, `guests`, `billing`, `publicMenu`, `analytics`) is a vertical slice of `domain/ → application/ → infra/ → ui/` plus a public `index.ts` barrel. Cross-module imports go through the barrel only — never deep-import another module's internals (e.g. `modules/inventory/index.ts`, never `modules/inventory/infra/...`).
- **Repository pattern**: define ports (interfaces) in `domain/`, provide both `*.http.ts` and `*.memory.ts` implementations in `infra/` so the UI runs against an in-memory mock before/independent of the backend; `application/` hooks depend only on the interface.
- **Pure domain math**: recipe cost/margin/usage (`recipes/domain/cost.ts`), billing totals (`billing/domain/totals.ts`), and stock depletion (`inventory/domain/depletion.ts`) are pure functions, unit-tested, never computed inside components.
- **i18n/RTL-ready**: ships LTR/English but must be Farsi/RTL-ready — use logical CSS props (`margin-inline`, `inset-inline`, etc.) and a `dir` attribute, never hardcoded `left`/`right`.
